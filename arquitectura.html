Arquitectura, Patrones y Estándares

La base técnica que sostiene SimFac: arquitectura por capas, patrones de diseño y convenciones de codificación.

🏗️ Arquitectura del Sistema

Vista por Capas y Clientes por Servidor

🖥️ Capa de Presentación (Cliente) Fase 1

HTML5 + CSS3

Interfaces: Panel Administrador (dashboard, CRUD), Menú Comensal (QR → catálogo → carrito → pedido). Presentación visual sin lógica de negocio en el cliente.

▼   fetch() / REST API   ▼

⚙️ Capa de Lógica de Negocio (Servidor) Fase 1

PHP + JavaScript (Lógica de Negocio)

Endpoints: /auth, /productos, /categorias, /pedidos, /mesas, /usuarios. Toda la lógica de validación, autenticación (JWT) y procesamiento de datos se ejecuta del lado del servidor.

▼   Consultas SQL   ▼

🗄️ Capa de Persistencia (Datos) Fase 1

MySQL — Base de Datos Relacional

8 tablas normalizadas (3FN). VARCHAR para roles y estados. Stored procedures para lógica compleja de pedidos. Triggers para auditoría.

❓ ¿Por qué esta arquitectura?

Cada decisión técnica fue tomada con base en los recursos disponibles, el nivel formativo y las necesidades reales del proyecto:

🔌 Arquitectura Cliente-Servidor con capas

  • Separación de responsabilidades: El frontend, backend y base de datos son independientes, permitiendo modificar una capa sin afectar las demás.
  • Escalabilidad: Se pueden agregar nuevos endpoints o interfaces sin reestructurar el sistema completo.
  • Aprendizaje: Refleja la arquitectura que se usa en la industria real, preparando al estudiante para entornos profesionales.

🌐 API REST

  • Estadoless: Cada petición contiene toda la información necesaria, facilitando la escalabilidad horizontal.
  • Estándar de la industria: Es el patrón más utilizado para comunicación cliente-servidor en la web moderna.
  • Compatibilidad: Permite que el mismo backend sirva tanto a la web como a una futura app móvil sin cambios.

🗄️ MySQL

  • Datos relacionales: Los pedidos, productos y usuarios tienen relaciones naturales que se modelan mejor con SQL.
  • ACID: Garantiza la integridad de las transacciones (un pedido no puede quedar a medio registrar).
  • Disponibilidad: Es gratuito, de código abierto y ampliamente documentado en español.

🌐 JavaScript en el servidor

  • Lógica centralizada: JavaScript se ejecuta en el servidor (PHP/Node.js) para procesar validaciones, cálculos y reglas de negocio, garantizando que el cliente no pueda manipular la lógica del sistema.
  • Cliente ligero: El navegador solo recibe HTML+CSS estático y realiza peticiones al servidor mediante fetch(), sin ejecutar lógica de negocio local.
  • Seguridad: Al no tener lógica en el cliente, se evita que usuarios malintencionados manipulen el comportamiento de la aplicación.
🧩 Patrones de Diseño

¿Por qué MVC y Singleton? Ambos patrones se complementan dentro del contexto del proyecto formativo:

📐 MVC (Model-View-Controller)

  • Separación de responsabilidades: El Controller recibe la petición HTTP, el Model gestiona la base de datos y la View genera la respuesta HTML. Cada capa se puede modificar de forma independiente.
  • Estándar de la industria: Es el patrón más enseñado y utilizado en desarrollo web, lo que facilita la evaluación por parte de los instructores y la comprensión del código.
  • Escalabilidad: Permite agregar nuevas vistas (mesero, cajero) sin modificar la lógica del modelo existente.

🔑 Singleton (Conexión a BD)

  • Efficiencia: Mantiene una única instancia de conexión a MySQL durante todo el ciclo de vida del servidor, evitando abrir y cerrar conexiones en cada petición.
  • Control de recursos: En un entorno de hosting compartido (como el que usa SENA), limitar el número de conexiones simultáneas es crítico para no saturar el servidor.
  • Consistencia: Garantiza que todas las operaciones compartan la misma configuración de conexión y codificación.
📐

MVC (Model-View-Controller)

Separación entre datos (Model), interfaz (View) y lógica (Controller) en el backend. Cada ruta del API actúa como Controller, manipula Models y responde al cliente.

📦

Repository

Capa de abstracción entre el Controller y la base de datos. Centraliza todas las consultas SQL, facilitando pruebas y cambios de motor de base de datos.

🔑

Singleton

Una única instancia de la conexión a la base de datos durante todo el ciclo de vida del servidor, evitando abrir y cerrar conexiones innecesariamente.

📡

Observer

Para actualizaciones en tiempo real: cuando el estado de un pedido cambia, se notifica automáticamente al panel del administrador y al módulo del mesero.

📏 Estándares de Codificación
  • Nombres: snake_case para tablas y columnas, sin espacios ni caracteres especiales.
  • Precios: DECIMAL(10,2) para evitar problemas de punto flotante.
  • Contraseñas: Hash con bcrypt (salt rounds = 10), nunca almacenadas en texto plano.
  • Auditoría: Triggers en tablas críticas que registran cambios en logs_auditoria.
  • Stored Procedures: Para la inserción de pedidos (que afecta múltiples tablas de forma atómica).
  • Normalización: Hasta 3FN, sin datos redundantes.
  • JavaScript: ES6+, camelCase para variables y funciones.
  • HTML: Semántico (header, main, section, nav, footer).
  • CSS: Variables CSS para el sistema de diseño, Flexbox y Grid para layouts.
  • Responsivo: Mobile-first, probado en 320px, 768px y 1440px.
  • Fetch: Todas las peticiones al API con manejo de errores (try/catch) y feedback visual.
  • Accesibilidad: Labels en formularios, alt en imágenes, contraste WCAG AA.
  • Lenguaje: JavaScript (Node.js/Express) o PHP — según disponibilidad del servidor.
  • Endpoints: Nombres en plural y snake_case (/api/ordenes, /api/categorias_productos).
  • Códigos HTTP: Uso correcto (200, 201, 400, 401, 404, 500).
  • Autenticación: JWT con expiración de 8 horas para la sesión del administrador.
  • Validación: Sanitización de inputs en servidor, nunca confiar en la validación del cliente.
  • CORS: Configurado para permitir solo el dominio del frontend.
